# 형태소쪼개기 (CLAUDE.md)

이 문서는 구현 작업의 상위 규칙이다. 아래 원칙과 충돌하는 제안·기능 추가는 하지 않는다.

## 프로젝트 정의

글을 읽으면서 **사용자가 직접** 형태소 단위로 사선을 그어 분해하는 도구.
프로그램은 절단면 처리(그리기·복원 표시·주석·한자 조회)만 담당한다.

## 절대 규칙 — 구현 중 어기지 말 것

1. **자동 형태소 분석 기능을 만들지 않는다.** 제안도 하지 않는다. 문장을 넣으면 자동으로 쪼개주는 기능은 이 프로젝트의 목적과 정반대다. 쪼개는 판단은 100% 사용자의 몫이다.
   - **범위 명확화 (2026-07-13, 유저 직접 확인)**: 이 규칙은 **쪼개는 과정**에 대한 것 — 도마에서 사선 긋기 자체를 자동화하거나 제안하지 않는다는 뜻. 유저가 다 쪼갠 **결과 이후**에 실제 형태소분석기와 대조해서 "정답"을 보여주는 건 별개(허용) — 아래 "정답 비교" 참고.
2. **원문은 불변이다.** 분해가 원문 텍스트를 변형·재배열하지 않는다. 칼금과 첨자는 원문 위에 얹힌다 (교정지 방식).
3. **읽기 우선.** 읽는 흐름을 깨는 인터랙션·레이아웃 변경을 임의로 하지 않는다.
4. **요청받지 않은 기능을 추가하지 않는다.** 온보딩, 튜토리얼, 공유 버튼, 다크모드 등 일반적 "있으면 좋은 것"도 요청 전에는 만들지 않는다. 이 도구의 사용자는 제작자 본인 1명이다.

## 화면 구조 — 2열 (데스크톱 기준)

**왼쪽 열 (주 작업 영역):**
- **상단: 다음 문장** — 바로 다음 타자 1개만 표시. 시각적으로 저순위 처리.
- **하단: 도마** — 현재 작업 문장 1개. 칼질이 일어나는 유일한 영역.

**오른쪽 열: 글 전문 패널** — 글 전체 표시. 현재 도마에 오른 문장의 위치가 전문 안에서 식별 가능해야 한다. 분해 완료된 문장은 전문 안에서 완료 표시를 남기는 방향 검토 (감상 뷰 역할 흡수 가능).

문장 완료 → 다음 문장이 도마로 이동, 전문 패널의 현재 위치 갱신.

**반응형:** 화면이 좁아지면 전문 패널이 아래로 내려간다 (1열 전환). 모바일에서는 상단 "다음 문장" 칸이 문맥 유지를 담당하고, 전문은 부록 취급.

## 인터랙션

| 행위 | 의미 | 대상 |
|---|---|---|
| 드래그로 사선 긋기 | 자르기 (사용자 판단) | 한글 형태소 경계 |
| 클릭/탭 | 조회 | 한자어 글자 (부수/구성 정보) |

- 한자어 내부는 긋기 대상이 아니다. 클릭 조회 전용.
- 마우스 드래그와 터치 스와이프 모두 지원. 데스크톱 우선, 모바일은 반응형 대응.

## 긋기의 두 가지 결과

1. **일반 절단** (예: 사람/의) — 빨간 사선만 남는다.
2. **복원 발생** (예: 예상치 → 예상+하+지) — 파란 사선 + 원문 위에 파란 첨자로 복원형이 나타난다. 첨자는 바뀐 부분만 표기한다 (전체 복원형 아님).
   - 복원이 펼쳐지는 순간의 애니메이션이 이 도구의 핵심 경험이다. 구현 우선순위 최상위.

## 시각 문법

### 시끄러운 층 (사건)
- **빨간 사선** = 절단
- **파란 사선 + 파란 첨자(어절 위)** = 복원

### 조용한 층 (정보)
- **어미 라벨** (ㄹ지, ㄴ다 등) = 복원이 아닌 이름표. 복원 첨자와 시각적으로 구분해 한 톤 낮춘다 (톤은 디자인 파일 따름).
- **어원 주석** (예: 지붕 → 어원: 집+웅) = 화석 표시. 어절 아래 배치.

### 선 스타일
- **사선은 깔끔한 직선으로 그린다.** 손그림·연필 질감·불규칙 흔들림 효과를 넣지 않는다.
- 줄간격은 넉넉하게. 첨자(위)와 주석(아래)이 본문 행과 충돌하지 않아야 한다. 긴 문장이 3~4줄로 꺾이는 경우가 기준.

### 반복 처리
- 같은 형태소가 반복 등장해도 매번 표기한다. 생략하지 않는다.

## 데이터 스키마

노드 (재귀, 전 층위 동일 구조):

```
{
  surface: string,      // 표면형 "했다"
  parts: Node[],        // 분해 [하, 였, 다]
  annotation?: {        // 선택
    type: "복원" | "어원" | "라벨" | "미확정" | "서사" | "허위",
    text: string
  },
  level: "글" | "문단" | "문장" | "어절" | "형태소"
}
```

- 위층(글·문단·문장)은 내비게이션 역할만 한다. 자동 문장 분리 로직에 과투자하지 않는다 (사용자가 경계를 수정할 수 있으면 충분).
- **축약 복원 테이블**: 사용자가 등록하는 데이터 (했→하였, 예상치→예상하지, 그게→그것이, 어떻게→어떠하게, 위해서→위하여서 …). 하드코딩 사전을 크게 만들지 말고 사용자 입력으로 쌓이는 구조로.
- **판정 아카이브**: 사용자의 판정 기록 저장. 같은 표면형 재등장 시 이전 판정을 참조로 제시 (자동 적용은 하지 않는다 — 제시만). 동형이의 존재 주의: 되-(접두사) vs 되-(어간).

## 사이트 구성

- **본관** — 분해 작업 화면 + 감상 뷰.
- **정답 비교** — 본관 화면 스크롤 다운 시 나오는 풀스크린 페이지(2026-07-13 추가). 유저가 쪼갠 전문과 대조할 수 있도록 실제 형태소분석기(kiwipiepy) 결과를 보여줌. 절대규칙 1 예외 — "쪼개는 과정"이 아니라 "결과 이후 대조"라 허용됨.
- **진열장** — 별도 페이지. 분해가 안 되는 단어들의 표본 전시 (짭새·짤·길막 등). 본관 완성 후 작업. 지금은 만들지 않는다.

## 기술 전제

- 정적 웹 (서버 없음) 원칙. **배포는 2026-07-14부터 GitHub Pages** — 이 저장소(main 브랜치)에 push하면 https://mrsbarnes.github.io/jjogaegi/ 에 자동 반영됨. intpdiary Flask 서버의 static/morpheme/ 서빙 방식(심볼릭 링크)은 이 날짜에 완전히 폐지됨 — 이제 GitHub Pages가 유일한 배포처.
- **예외 1**: 맞춤법 교정 기능(2026-07-13 추가)은 다음(Daum) 맞춤법 검사기를 CORS 없이 호출하기 위해 intpdiary Flask에 `/api/morpheme/spellcheck` 중계 라우트를 둠 (`routes/morpheme.py`, intpdiary 저장소). GitHub Pages는 정적 사이트라 이 백엔드를 대체할 수 없음 — script.js는 절대 주소(`https://intpdiary.duckdns.org/api/morpheme/spellcheck`)로 호출하고, intpdiary의 `app.py`에 이 출처(`https://mrsbarnes.github.io`)를 허용하는 CORS 헤더가 추가돼 있음(2026-07-14).
- **예외 2**: 정답 비교 기능(2026-07-13 추가)은 intpdiary Flask에 kiwipiepy(순수 파이썬/C++ 형태소분석기, 자바 불필요)를 `pip install --break-system-packages`로 설치하고 `/api/morpheme/analyze` 라우트를 둠(intpdiary 저장소). 마찬가지로 절대 주소 + CORS로 GitHub Pages에서 호출. 서버 재시작 시 모델 로딩으로 메모리 사용량이 커짐(약 500MB+) — 재시작 후 정상 기동 여부 확인할 것.
- 위 두 예외(백엔드 API 자체)는 intpdiary 저장소의 `routes/morpheme.py`에 그대로 남아있고 건드리지 않는다 — 여기(jjogaegi)서는 프론트엔드 코드만 관리한다. 위 두 예외 외에는 서버 의존을 확장하지 않는다.
- 저장은 로컬 우선(localStorage). 동기화가 필요해지면 그때 논의 (선제 구현 금지).
- 외부 형태소 분석 라이브러리를 **쪼개는 과정**에 도입 금지 (절대 규칙 1 참조). kiwipiepy는 "결과 이후 정답 비교"용으로만 쓰고, 도마의 사선 긋기 로직에는 절대 연결하지 않는다.
